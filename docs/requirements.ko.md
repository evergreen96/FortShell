# AI IDE 요구사항

## 1. 제품 정의

### 1.1 문제 정의

기존 IDE는 AI를 에디터에 붙는 보조 기능으로 취급한다. 우리가 만들고자 하는 제품은 다르다. 이 제품에서 IDE는 여러 AI 에이전트와 CLI 도구를 안전하게 실행하고, 관찰하고, 무엇을 볼 수 있는지 통제하는 실행 환경이어야 한다.

핵심 요구사항은 단순히 "좋은 자동완성"이 아니다. 핵심은 다음이다.

- 각 AI 세션이 어떤 파일을 읽고, 쓰고, 검색하고, 실행할 수 있는지 프로젝트 단위로 제어할 수 있어야 한다.
- Claude Code, Codex CLI, Gemini CLI, OpenCode 같은 에이전트 CLI를 강하게 격리할 수 있어야 한다.
- 여러 터미널을 동시에 운영하고, 터미널과 AI 세션 간 조율이 가능해야 한다.
- 정책이 바뀌면 기존 컨텍스트가 남지 않도록 즉시 새 세션을 만들어야 한다.
- read/write/exec 횟수, 감사 로그, 차단 이벤트, 지연 시간, 비용 같은 관측 정보가 자세히 제공되어야 한다.

### 1.2 제품 가설

사용자 경험은 일반 IDE와 비슷하되, 레이아웃의 중심이 코드 에디터가 아니라 터미널이다.

- 프로젝트(폴더)를 열면 해당 폴더가 워크스페이스가 되고, 그 범위 안에서만 정책이 적용된다.
- 왼쪽 파일 트리에서 파일·폴더마다 체크박스로 AI에게 노출할지 여부를 선택할 수 있다. 체크 해제된 항목은 에이전트의 CLI 검색, 읽기, 목록 어디에도 나타나지 않는다.
- 메인 영역은 멀티 터미널이 차지한다. 여러 터미널 탭을 동시에 운영할 수 있다.
- 코드 편집기는 파일 트리에서 파일 아이콘을 클릭하면 오른쪽 패널로 열린다. 에디터는 보조 도구이며 터미널이 주 작업 공간이다.
- 정책(가시성)을 변경하면 활성 AI 세션은 즉시 무효화되고 새 세션이 시작된다.

### 1.3 사용자

- AI CLI를 많이 사용하는 개인 개발자
- 하나의 저장소 안에서 로컬 프라이버시 경계가 필요한 소규모 팀
- 감사 가능한 AI 보조 개발 환경이 필요한 조직

## 2. 제품 원칙

- 기본은 차단이다. 명시적으로 허용하지 않으면 접근할 수 없어야 한다.
- 정책은 프롬프트가 아니라 런타임이 강제해야 한다.
- 숨겨진 파일은 "보지 않도록 권고"하는 수준이 아니라 실제로 보이지 않아야 한다.
- 세션은 폐기 가능한 단위여야 하며, 정책이 바뀌면 기존 세션은 무효화되어야 한다.
- 관측성은 부가 기능이 아니라 제품의 핵심 화면이어야 한다.
- 보안 경계는 테스트 가능하고 벤치마크 가능해야 한다.
- 새로운 에이전트 CLI를 추가할 때 코어를 다시 설계하지 않아야 한다.

## 3. 기능 요구사항

### 3.0 IDE 기본 기능

#### 3.0.1 프로젝트 열기

- 사용자가 폴더를 열면 해당 폴더가 워크스페이스 루트가 된다.
- 워크스페이스 루트를 기준으로 정책, 세션, 감사 범위가 결정된다.
- 프로젝트 설정(`.ai-ide/`)이 없으면 기본 정책으로 초기화한다.

#### 3.0.2 파일 트리

- 왼쪽 패널에 워크스페이스의 파일·폴더 트리를 표시한다.
- 각 항목에 체크박스가 있어 AI 에이전트에 대한 노출 여부를 토글할 수 있다.
- 체크 해제된 항목은 에이전트의 모든 접근(list, search, read, glob, index)에서 제외된다.
- 체크박스 상태 변경은 정책 변경으로 취급되며, 활성 AI 세션을 무효화하고 새 세션을 시작한다.
- 사용자 자신은 체크 해제 여부와 관계없이 모든 파일을 볼 수 있다.

#### 3.0.3 멀티 터미널 (메인 영역)

- 화면 중앙을 터미널이 차지한다.
- 여러 터미널 탭을 동시에 열고 전환할 수 있다.
- 각 터미널은 독립된 상태, 히스토리, 작업 디렉터리, 환경 변수를 가진다.
- PTY 동작, 리사이즈, 색상, 대화형 프롬프트 등 터미널 본래 기능을 보존한다.

#### 3.0.4 코드 편집기 (보조 패널)

- 파일 트리에서 파일을 클릭하면 오른쪽 패널에 코드 편집기가 열린다.
- 편집기는 기본적인 코드 편집 기능(구문 강조, 검색, 줄 번호 등)을 제공한다.
- 편집기는 보조 역할이며, 터미널이 주 작업 공간이다.

### 3.1 워크스페이스 정책 모델

**제품 계약**: workspace 내부에는 deny-list를 적용하고, workspace 외부는 정책 대상이 아니며, 내부 메타데이터(`.ai-ide`, `.ai_ide_runtime`)는 항상 숨긴다.

- IDE는 프로젝트별 파일, 디렉터리, 파일 유형 단위의 가시성 규칙을 지원해야 한다.
- 차단된 경로는 에이전트에게 제공되는 list, search, glob, completion, index, read 결과 어디에도 나타나면 안 된다.
- 에이전트가 `find`, `git`, `python`, `node`, 셸 내장 명령, 심볼릭 링크, 압축 파일, 자식 프로세스 같은 우회 경로를 시도해도 차단된 경로는 계속 막혀 있어야 한다.
- 정책은 최소한 다음을 지원해야 한다.
  - 경로 glob 기반 allow/deny
  - 작업 단위 allow/deny: read, write, execute, search
  - 에이전트별 또는 세션별 override
  - 만료 시간이 있는 임시 승인
- 정책 평가는 결정적이어야 하며 버전이 관리되어야 한다.

### 3.2 에이전트 런타임

- IDE는 여러 에이전트 CLI를 동시에 실행할 수 있어야 한다.
- 각 에이전트 실행은 정책 스냅샷을 가진 격리된 세션 안에서 이루어져야 한다.
- 런타임은 서로 다른 에이전트의 동작을 공통 추상화로 정규화해야 한다.
  - launch
  - stop
  - stdout/stderr 스트리밍
  - 도구 호출 텔레메트리
  - 정책 차단 보고
  - 종료 상태와 크래시 원인
- 런타임은 대화형 에이전트 CLI와 MCP 같은 프로토콜 기반 도구를 모두 지원해야 한다.

### 3.3 터미널–에이전트 연동

- 터미널과 터미널, 터미널과 에이전트 간 구조화된 이벤트 채널을 통해 명시적 메시징이 가능해야 한다.
- 각 터미널은 세션 소유권을 가지며, 에이전트 세션과 연결될 수 있다.

### 3.4 세션 생명주기

- 정책이 바뀌면 자동으로 새 세션을 생성해야 한다.
- 기존 세션은 stale 상태로 표시하고 더 이상 권한 있는 접근을 할 수 없게 해야 한다.
- 세션 기록에는 최소한 다음이 포함되어야 한다.
  - session ID
  - policy version
  - runner type
  - allowed roots
  - environment profile
  - agent identity
  - 시작/종료 시각
- 사용자는 왜 세션이 교체되었는지 확인할 수 있어야 한다.

### 3.5 감사와 관측성

- IDE는 read, write, search, command execution, denial, network attempt, session rotation을 모두 카운트해야 한다.
- 차단된 모든 동작에는 reason code가 있어야 한다.
- 감사 데이터는 프로젝트, 세션, 에이전트, 터미널, 파일, 시간 범위로 조회할 수 있어야 한다.
- 제품은 다음 같은 고신호 화면을 제공해야 한다.
  - 에이전트가 무엇을 읽었는지
  - 무엇을 썼는지
  - 무엇이 막혔는지
  - 정책 변경 후 무엇이 달라졌는지
- 상위 도구가 정보를 제공하는 경우 비용과 토큰 사용량도 지원해야 한다.

### 3.6 리뷰와 쓰기 반영

- AI가 만든 쓰기 작업은 어떤 세션에서 발생했는지 추적 가능해야 한다.
- 시스템은 선택적으로 staged writeback을 지원해야 한다.
  - 먼저 샌드박스 projection에 기록
  - diff 검토
  - 승인된 변경만 host workspace에 반영
- 머지 충돌과 host drift는 명시적으로 드러나야 한다.

### 3.7 프로젝트 모델

- 프로젝트 설정은 휴대 가능하고 소스 제어에서 검토 가능해야 한다.
- 정책은 `.ai-ide/policy.json` 같은 프로젝트 파일에 저장되어야 한다.
- 세션 로그와 감사 데이터는 기본적으로 소스 제어 밖에 저장되어야 한다.
- 팀은 공유 정책 프리셋을 정의할 수 있어야 한다.

## 4. 비기능 요구사항

### 4.1 보안

- 보안 경계는 모델의 순응성에 의존하면 안 된다.
- 제품은 에이전트가 호스트 파일시스템에 직접 접근하지 못하는 strict mode를 지원해야 한다.
- 시크릿 처리는 명시적이어야 한다. 환경 변수, 자격 증명, 토큰에는 정책 제어와 감사 추적이 있어야 한다.
- 런타임은 prompt injection에 의한 도구 남용, symlink traversal, path escape, child-process escape를 완화해야 한다.

### 4.2 유지보수성

- 핵심 정책 평가는 UI 의존성이 없는 작고 테스트하기 쉬운 모듈에 있어야 한다.
- OS별 샌드박스 코드는 어댑터 뒤로 격리되어야 한다.
- 에이전트 어댑터 API는 안정적이고 버전 관리되어야 한다.
- persistence, runtime, terminal, UI 관심사는 명확히 분리되어야 한다.

### 4.3 테스트 가능성

- 모든 보안 규칙에는 자동 회귀 테스트가 있어야 한다.
- 제품은 fixture 저장소를 활용한 결정적 통합 테스트를 지원해야 한다.
- 벤치마크 작업 부하는 머신과 CI 환경에서 재현 가능해야 한다.

### 4.4 성능

- 터미널과 에이전트가 출력을 스트리밍하는 동안에도 UI는 반응성을 유지해야 한다.
- 정책 검사 오버헤드는 일반적인 read/search 경로에서 낮아야 한다.
- 큰 저장소에서도 사용 가능해야 한다.
- 정책 변경 후 세션 교체는 즉각적으로 느껴질 만큼 빨라야 한다.

### 4.5 신뢰성

- 에이전트가 크래시하더라도 IDE 셸 전체가 함께 죽으면 안 된다.
- 런타임은 멈춘 터미널, 깨진 PTY, 부분적인 writeback 실패에서 깨끗하게 복구해야 한다.
- 감사 데이터는 애플리케이션 재시작 후에도 유지되어야 한다.

## 5. 위협 모델

### 5.1 범위에 포함되는 위협

- 저장소 파일이나 터미널 출력으로 인한 prompt injection
- 다른 도구나 subprocess를 통한 무단 읽기
- 차단된 경로에 대한 무단 쓰기
- path traversal과 symlink escape
- 네트워크를 통한 데이터 유출
- 정책 변경 후 stale session 재사용
- 서드파티 에이전트 CLI나 플러그인에 따른 공급망 위험

### 5.2 범위 밖 위협

- 커널 또는 하이퍼바이저 제로데이
- 이미 사용자 전체 권한으로 실행 중인 악성코드
- 물리적 접근 공격

### 5.3 보안 가정

- 데스크톱 셸 자체는 신뢰한다.
- 에이전트 CLI는 신뢰하지 않거나 부분적으로만 신뢰한다.
- 렌더러와 웹 UI 코드는 코어 런타임보다 낮은 신뢰 수준으로 본다.
- 승인된 projection 밖의 호스트 파일은 에이전트가 접근할 수 없는 자산으로 취급한다.

## 6. 성공 기준

### 6.1 MVP

- 최소 두 개 이상의 외부 AI CLI를 공통 런타임으로 실행할 수 있어야 한다.
- read, write, search에 대한 allow/deny 가시성 제어가 동작해야 한다.
- 멀티 터미널을 제공해야 한다.
- 정책 변경 시 세션이 자동으로 교체되어야 한다.
- 감사 로그와 메트릭이 로컬에 영속화되어야 한다.
- 일반적인 탈출 시도에 대한 초기 보안 회귀 테스트를 통과해야 한다.

### 6.2 프로덕션 준비 상태

- 지원 플랫폼에서 strict sandbox mode가 제공되어야 한다.
- 검토 가능한 writeback 워크플로우가 있어야 한다.
- 정책 파일 포맷이 안정화되어야 한다.
- 대형 저장소용 재현 가능한 벤치마크 스위트가 있어야 한다.
- 크래시 복구와 로그 보존이 갖춰져야 한다.

## 7. 열려 있는 제품 결정 사항

- strict mode를 모든 플랫폼에서 초기 출시부터 요구할지, 단계적으로 도입할지
- host write를 직접 허용할지, 항상 staged writeback만 통하게 할지
- 네트워크 접근을 모든 에이전트에서 기본 차단으로 둘지, 프로필별로 설정할지
- 프로젝트 정책 중 얼마나 많은 부분을 소스 제어에 공유하고, 어떤 부분을 사용자 로컬 override로 둘지
